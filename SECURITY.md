# 安全策略

## 报告漏洞

请不要在公开 Issue 中披露可直接利用的漏洞。优先通过 GitHub Security Advisory 私下报告，并提供：

- 受影响版本或提交；
- 部署方式和 dae 版本；
- 可复现步骤；
- 预期影响；
- 已知缓解措施。

## 安全边界

`kdae-panel` 是具有高权限的系统管理服务。当前发布模型以 root 运行，但将操作限制为固定的 dae、systemctl 和 journalctl 参数，不提供终端或任意 Shell API。

主要防护包括：

- 管理员密码使用 Argon2id，随机盐值独立保存；
- 登录令牌只在创建时返回到 HttpOnly Cookie，数据库只保存 SHA-256 摘要；
- 会话使用绝对过期时间，修改密码会注销旧会话；
- 非只读请求要求会话专属 CSRF 令牌并校验 Origin；
- 登录失败按来源地址限速；
- API 请求体、命令输出和日志条数都有上限；
- systemd 服务采用能力白名单和文件系统保护；
- 配置及备份默认使用 0600 权限，数据目录使用 0700。

## 部署要求

- 默认保持 `127.0.0.1` 监听；
- 远程访问必须使用 HTTPS 反向代理或 SSH 隧道；
- HTTPS 部署必须设置 `KDAE_PANEL_SECURE_COOKIE=true`；
- 反向代理必须保留原始 Host；
- 不要让非管理员写入 `/etc/kdae-panel`、`/var/lib/kdae-panel`、dae 配置或面板二进制；
- 定期更新面板、dae、Go 工具链、Node 构建依赖和操作系统；
- 备份可能包含订阅地址和节点凭据，分享前必须脱敏。

## 已知边界

- 首次初始化采用“首个成功请求创建管理员”的模式。公开网络监听前必须先在本机完成初始化。
- 面板不能防止已拥有 root 权限的本机攻击者读取数据库、替换二进制或直接修改 dae。
- `sysdump` 和 dae 校验错误可能包含系统信息，只有已登录管理员可以读取。
- 当前没有稳定的 dae 运行时连接 API，因此不会读取或暴露内部 eBPF Map。
